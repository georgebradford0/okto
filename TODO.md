Today `exec_create_agent_for_parent` always reads *lair's* `mcp.json` (`octo_core::mcp::load_mcp_configs()`) when the `mcp` field is omitted — regardless of whether the spawn was triggered by the operator or by another agent. That means per-agent edits don't propagate: if parent agent A has had `octo mcp add --agent A some-tool` applied to its `mcp.json`, then when A calls `spawn_agent` to create child B, B still inherits from lair (and won't get `some-tool`). - [ ] Replace `GH_TOKEN` PAT-in-env with a GitHub App + installation tokens. Today operators set `GH_TOKEN=ghp_…` via `octo init --env` and it's forwarded verbatim to every child — a long-lived classic PAT with `repo`-wide scope that lands in `docker inspect` on lair and every child, with no per-agent or per-repo scoping and no expiry. Replace with the App-installation-token flow GitHub Actions uses internally: (a) one-time setup via the GitHub App manifest flow (`octo gh-app setup` opens a URL, operator clicks Install on github.com, picks repos, redirects back; CLI captures App ID + private-key PEM into `config.json`); (b) lair becomes a token broker — signs a short RS256 JWT with the App private key (~10 min), exchanges it at `POST /app/installations/{id}/access_tokens` for a ~1-hour installation token scoped to the App's selected repos; (c) `exec_create_agent` injects a freshly-minted installation token instead of a PAT, and lair exposes a `request_gh_token` RPC over the existing Noise tunnel so long-running children can refresh on demand. Wins: short-lived tokens (blast radius ~1h instead of "until manually revoked"), per-repo scoping at install time, no manual PAT pasting, no `--env GH_TOKEN=…` opt-in. Implementation: ~200 lines of Rust + one `jsonwebtoken` dep + a CLI subcommand for the manifest flow. As an interim cheap stopgap consider rewriting HTTPS git URLs to SSH form inside `exec_create_agent` and mounting lair's existing `/data/ssh_id_ed25519` into children (covers the clone path with zero new auth — but doesn't fix `gh` CLI calls, so it's a partial answer at best).
- [ ] Front the push-notification relay with CloudFront + AWS WAF to absorb L7 floods / bad-signature spam before they reach the t4g.nano. Today the relay box (`octorelay.directto.link` → A → Elastic IP `100.20.194.81`, EC2 `i-02a59f2d7674a4684`, t4g.nano in us-west-2b, SG `sg-026004082bfcfd94e` opening 80+443 to `0.0.0.0/0`) is bare on a public IP; TLS is terminated on-box by Caddy with a Let's Encrypt cert. A flood of new TLS connections — not the cheap Ed25519 verify — is what drains the nano's burst CPU credits. The Cloudflare option (see STAGED_TODO.md) was rejected because the `directto.link` zone lives in Route53 and Cloudflare's proxy needs to be authoritative DNS, forcing either a full zone migration (blast radius across `admin`/`lenguas`/`plog`/`upworkprototypes` + SES email records) or a hostname change (needs a CLI/lair release for `DEFAULT_RELAY_URL`). CloudFront avoids both: keep the zone in Route53 and just alias `octorelay.directto.link` → the CloudFront distribution — no NS change, hostname unchanged. Steps: (a) create a CloudFront distribution with the EC2 box as a custom origin; (b) attach an AWS WAF web ACL with rate-based rules on `/notify` and `/register` plus an AWS managed rule group; (c) repoint the Route53 record to the distribution; (d) **only after CloudFront is serving**, lock SG ingress on 80+443 — replace `0.0.0.0/0` with the managed prefix list `com.amazonaws.global.cloudfront.origin-facing` so attackers can't bypass the edge by hitting the EIP directly (doing this before the cutover cuts the relay off); (e) make Caddy trust `CF-Connecting-IP`-equivalent (`X-Forwarded-For` from CloudFront) for real client IPs in logs. Cost: CloudFront ~$0 (perpetual free tier easily covers a push relay's traffic); AWS WAF ~$8/mo ($5 web ACL + ~$1/rule + $0.60/M requests) — the only new spend, and the price of skipping the DNS migration. Shield Standard is free and already on. Whole thing is doable via the AWS CLI; no external credentials. Note the relay is low-value and recoverable (SQLite rebuilds itself — devices re-register idempotently on next chat mount, and lair's `notify()` is best-effort), so an outage is degraded-not-broken; this is worth doing but not urgent.

- [ ] Remote agent update logic for `octo lair update`. Today `octo lair update` pulls a new lair image and recreates the local container, but **remote agents are untouched** — they keep running the old `octo-lair` image on their VMs, so after a lair version bump the fleet is version-skewed (a mismatched remote agent could break the Noise/wire proxy). `octo reload` is fine — it reuses the same image, no version change. Mechanism: lair already owns remote-agent control over SSH (`register_remote_agent` does `docker pull` + `systemctl restart octo-agent` + `agent-info.json` confirm), so add a management endpoint (e.g. `POST /agents/update-remote` or per-agent `POST /agents/:name/update`) the CLI calls — the outgoing lair, still alive with the SSH key, updates each remote VM and writes the new `binary_version` into the bind-mounted registry before the container swap. Open decisions: (1) image tag — point remote systemd units at `:latest` (simple `pull` + `restart`) vs keep version-pinned tags and SSH-rewrite the unit's `ExecStart` + `daemon-reload` (explicit, stageable, makes `binary_version` truthful); (2) automatic vs opt-in — plain `octo lair update` could just print a version-skew report and gate remote updates behind `--include-remote`, since restarting a remote VM container has real downtime and SSH can fail; (3) per-agent best-effort failure handling so an unreachable VM doesn't abort the run. Also fold in: `octo lair update` currently leaves every *local* agent `Stopped` (killed with the container, never auto-restarted — see the startup adoption pass in `lair/src/lair.rs`), so auto-restart the local agents that were `Running` before the swap. Deferred for now — agents are assumed short/medium-lived, so revisit once the system has had real usage.
